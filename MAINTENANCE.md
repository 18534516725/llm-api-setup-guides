# 仓库维护手册

这份清单用于保持教程真实、链接可用、示例安全和文档站可部署。维护时优先修正会阻止用户完成配置的问题，再扩充新工具。

## 每次提交前

```bash
node --test tests/*.test.mjs
node scripts/validate-docs.mjs
.venv-docs/bin/mkdocs build --strict
git diff --check
```

确认没有真实 API Key、Cookie、内部渠道信息或私人日志。新增教程时使用 [`guides/教程模板.md`](./guides/教程模板.md)，并同步 `mkdocs.yml`、教程总目录和兼容性矩阵。

## 每周

- 查看“文档质量检查”“部署 GitHub Pages”和“每周外部链接巡检”的运行结果；
- 处理教程纠错和新增教程 Issue；
- 检查 Dependabot 提交的 GitHub Actions 更新；
- 抽查文档站首页、搜索、手机导航和 NexoToken 入口。

## 每月

- 按教程访问量和 Issue 频率抽查热门文档；
- 核对安装命令、设置入口、配置字段、模型能力和官方链接；
- 检查 README、在线站点、教程总目录和兼容性矩阵是否一致；
- 更新每篇发生变化教程的“最后核验”日期，未核验的页面不要只改日期。

## 产品大版本发布时

1. 阅读官方发布说明与迁移指南；
2. 在干净环境复现安装和最小请求；
3. 核对协议、Base URL 拼接方式、模型 ID、流式输出和工具调用；
4. 更新截图或界面路径时彻底遮住账户与密钥；
5. 在 `CHANGELOG.md` 记录用户可感知的变化；
6. 运行完整检查并确认 Pages 部署成功。

## 每季度

- 审查目录是否仍按用户场景清晰分组；
- 合并重复教程，标记停运或停止维护的产品；
- 检查仓库 Description、Homepage、Topics、社交预览和安全设置；
- 查看访问量、热门路径和来源，只据此确定核验优先级，不伪造流量或批量制造低质量页面；
- 复查工作流权限，保持最小权限。

## 发布故障处理

- 文档校验失败：按输出的文件、行号修正后本地重跑；
- Pages 构建失败：查看 `mkdocs build --strict` 的第一条 warning 或 traceback；
- 外部链接失败：先在浏览器核验，确认不是限流、登录页或短时故障后再修改；
- 部署成功但页面旧：确认 Actions 部署的提交 SHA 与 `main` 一致，并检查浏览器缓存；
- 密钥误提交：立即撤销密钥，按安全事件处理；仅删除 Git 文字不能恢复已经泄露的凭据。

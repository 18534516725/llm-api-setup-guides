# GitHub 中文 AI 教程站长期建设设计

**日期：** 2026-07-15

**状态：** 已确认

**仓库：** `18534516725/llm-api-setup-guides`

## 目标

把现有仓库建设成一个可持续维护、可被搜索引擎收录、对小白友好，同时能自然引导用户了解 NexoToken 的中文 AI API 教程中心。

成功标准：

1. 仓库根 README 能在一分钟内说明项目价值、覆盖范围、快速入口和服务入口。
2. GitHub Pages 提供独立文档站，具备站内搜索、清晰导航、移动端适配、深色模式、站点地图和规范链接。
3. NexoToken 在 README 与文档站首页首屏可见，但教程正文保持以解决问题为主，避免每段重复营销。
4. 新增或修改文档时，自动检查内部链接、示例语法、敏感信息和站点构建。
5. Issues、Pull Request、贡献指南和维护说明形成可重复执行的维护流程。
6. GitHub 仓库元数据、Topics、Pages 地址和社交预览图保持一致。

## 方案选择

采用“文档站优先”的方案：仓库 Homepage 字段指向 GitHub Pages，NexoToken 入口保留在 README 和文档站首页显眼位置。

没有采用“Homepage 直接跳转服务站”的原因是，教程站域名更符合仓库本身的内容语义，也更利于用户分享具体教程页面和搜索引擎理解站点结构。服务入口通过首屏按钮、顶部导航和教程末尾统一提示承接访问。

## 信息架构

文档站以 `guides/` 为内容根目录，按使用场景组织，而不是按模型厂商组织：

- 入门基础：API、协议、密钥、模型名、常见错误。
- 编程工具：Claude Code、Codex CLI、Qwen Code、Gemini CLI、Aider、Cline 等。
- 聊天客户端：Cherry Studio、Chatbox、LobeChat 等。
- 开发集成：SDK、Agent 框架、LangChain、LlamaIndex 等。
- 自托管平台：Dify、Open WebUI、RAGFlow 等。
- 自动化平台：n8n、Flowise、Langflow。
- 效率工具：翻译、笔记等。
- API 测试：Apifox、Postman 和命令行验证。

文档站首页负责分流，分类首页负责解释选择标准，具体教程负责从安装到验证和排错。导航中的名称使用中文，产品名保留官方写法，URL 路径继续使用稳定的英文短名，避免中文路径在分享和迁移时产生兼容问题。

## 推广位置与表达

NexoToken 使用三级曝光：

1. README 首屏：项目价值说明后提供“获取 API Key / 查看模型”主入口。
2. 文档站首页：首屏行动按钮和“开始前准备”卡片中展示服务入口。
3. 教程正文：统一在准备工作或文末放置一次中性提示，说明教程支持自定义 API 地址，不编造使用体验，不声称未经验证的价格或模型能力。

不在每个步骤插入广告，不伪装成第三方推荐，不使用虚假用户故事。所有价格、模型可用性和活动额度等易变化信息只链接到服务站实时页面，不在长期教程中写死。

## 技术架构

站点使用 MkDocs 与 Material for MkDocs：

- `mkdocs.yml`：站点信息、导航、主题、搜索、SEO 和扩展配置。
- `requirements-docs.txt`：固定文档构建依赖版本，保证本地与 GitHub Actions 一致。
- `guides/index.md`：文档站首页。
- `guides/assets/`：站点 Logo、社交预览等静态资源。
- GitHub Actions：质量检查与 Pages 部署分开运行。

Pages 工作流在 `main` 分支内容变更后构建并部署。Pull Request 只执行构建与质量检查，不执行部署。部署使用 GitHub 官方 Pages Actions，并只授予部署所需的最小权限。

## 搜索与 SEO

站点提供以下搜索信号：

- 每页唯一且描述准确的中文标题与摘要。
- 稳定的英文 URL、规范链接和站点地图。
- Open Graph 社交分享信息和统一封面图。
- README、文档首页、分类页之间的可爬取内部链接。
- 仓库 Description、Topics、Homepage 与页面内容一致。
- `robots.txt` 允许正常抓取，站点地图交由搜索引擎发现。

不承诺搜索排名，不使用关键词堆砌、隐藏文本、批量伪原创或垃圾外链。内容增长以真实教程覆盖、问题解决质量和持续更新为主。

## 自动质量检查

仓库增加一个只读校验脚本，职责限定为：

1. 扫描 Markdown 内部链接，确认目标文件和锚点存在。
2. 检查教程是否包含标题、适用场景、验证步骤和排错内容等最低结构。
3. 解析标记为 JSON 的代码块，检查语法。
4. 编译 Python 示例代码块，检查基础语法。
5. 检测疑似真实 API Key、Bearer Token 或私密配置，允许明确的占位符。
6. 校验导航引用的文档存在。

脚本使用 Node.js 标准库实现，不引入运行时依赖。测试先覆盖链接解析、占位符放行和敏感值拦截，再实现校验逻辑。GitHub Actions 同时运行脚本和 `mkdocs build --strict`。

外部链接单独使用每周定时任务检查，避免第三方网站暂时不可用阻塞每次提交。定时任务允许人工重跑，并对限流、登录页和反爬响应做保守处理。

## 社区与维护机制

补齐以下社区文件：

- Pull Request 模板：要求说明修改范围、验证方式和截图需求。
- Issue 配置：保留教程请求和纠错表单，关闭空白 Issue 或明确使用范围。
- Code of Conduct：采用 Contributor Covenant，明确社区行为边界。
- 维护指南：记录每周、每月和模型/工具大版本更新时的检查清单。
- 教程模板：统一目标读者、环境、配置、验证、常见错误、安全说明和更新时间。

仓库关闭未使用的 Wiki 和 Projects，把知识内容集中在版本控制内。Issues 保持开启，用于纠错和教程请求。合并策略以 Squash Merge 为主，并自动删除已合并分支；`main` 保持可由维护者直接更新，避免个人维护仓库被过重保护规则阻塞。

## 安全与隐私

- 所有示例只使用明显占位符，例如 `sk-your-key`、`https://your-api.example.com/v1`。
- 不提交真实密钥、Cookie、访问令牌、内部渠道标识或服务端配置。
- 自动化工作流使用 GitHub 默认令牌和最小权限，不新增长期密钥。
- 开启 Secret Scanning、Push Protection、Dependabot 安全更新和私密漏洞报告。
- 教程提醒用户不要截图或公开粘贴完整密钥。

## 错误处理与回滚

- Pages 构建失败时保留上一版成功部署，不发布半成品。
- 质量检查给出具体文件、行号和错误原因，便于本地复现。
- 外部链接失败不自动修改文档，只创建可审查的结果。
- 仓库设置通过 GitHub API 修改后立即重新读取确认；失败的设置单独报告，不影响已提交的文档文件。
- 所有建设变更集中为可审查的 Git 提交，可通过普通反向提交回滚。

## 验证与验收

提交前执行：

1. 校验脚本自身测试通过，并确认测试曾在功能缺失时正确失败。
2. 全仓文档质量检查通过。
3. `mkdocs build --strict` 成功且无警告。
4. 扫描仓库，确认无疑似密钥和私密配置。
5. 检查所有新增内部链接和导航入口。
6. 推送后确认远端提交与本地一致、Actions 已触发、Pages 返回成功页面。
7. 重新读取仓库 Description、Homepage、Topics、安全设置和社区健康信息。

## 后续维护节奏

- 每周：查看 Issue、失败工作流和失效外链。
- 每月：抽查热门教程的安装命令、界面路径和配置字段。
- 工具大版本发布时：优先更新对应教程并在 CHANGELOG 记录。
- 每季度：审查目录结构、搜索词覆盖和低价值重复内容。
- 每次新增教程：从模板创建，完成本地校验后再提交。
